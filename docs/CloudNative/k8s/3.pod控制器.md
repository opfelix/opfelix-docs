# pod控制器

>本文引用的连接，阅读请点击  
>[Kubenetes的工作负载均衡](https://kubernetes.io/zh-cn/docs/concepts/workloads/controllers/)  
>[编写Deployment规约](https://kubernetes.io/zh-cn/docs/concepts/workloads/controllers/deployment/#writing-a-deployment-spec)

kubenetes提供了几个内置的API来声明式(通过配置而非代码逻辑来管理系统状态的方法)管理工作负载(又称为控制器)及组件。
kubenetes应用以容器的形式在Pods中运行，但是直接管理单个 Pod 的工作量会非常的繁琐，例如，一个Pod失败了，你希望运行一个新的Pod来替代它，这个时候就可以使用 kubernetes API 来创建控制器对象，kubernetes control根据定义的控制器对象自动管理Pod集群。
内置的 API 分为：
- 用于管理工作负载的 API
	- Deployment：适合在集群上管理无状态应用工作负载， 其中Deployment中的任何Pod都是可互换的，可以在需要时进行替换。
	- ReplicationController(被Deployment替代)
	- ReplicaSet
	- StatefulSet：允许你管理一个或多个运行相同应用代码、但具有不同身份标识的 Pod。
	- DaemonSet：定义了在特定节点上提供本地设施的Pod， 例如允许该节点上的容器访问存储系统的驱动。
- 用于定义任务的 API
	- Job：表示一次性任务
	- CronJob：可以根据排期表重复执行。
## Deployment

用于管理运行一个应用负载的一组Pod，通常适用于不保持状态的负载(无状态)，为Pod和ReplicaSet提供声明式的更新能力。

### 创建Deployment将ReplicaSet上线

```yaml
# 创建了一个ReplicaSet，负责启动三个 `nginx` Pod
apiVersion: apps/v1 # 版本
kind: Deployment # 资源类型
metadata: # 元数据
  name: nginx-deployment # Pod名字，集群唯一
  labels:
    app: nginx
spec: # 期望
  replicas: 3 # 创建的Pod数量
  selector: # 必选，选择算符，必须匹配.spec.template.metadata.labels，否则请求会被API拒绝
    matchLabels: # 字段是{key,value}键值对映射
      app: nginx
  template: # 必选，模版，同Pod
    metadata:
      labels:
        app: nginx
    spec:
      containers: # 容器列表
      - name: nginx # 容器名称
        image: nginx
        ports: # 容器端口
        - containerPort: 80 
```

1. 创建`kubectl apply -f [file]`
2. 运行`kubectl get deployment`查看deployment时候创建
	- `NAME`：列出了名字空间中Deployment 的名称。
	- `READY`：显示应用程序的可用的“副本”数。显示的模式是“就绪个数/期望个数”。
	- `UP-TO-DATE`：显示为了达到期望状态已经更新的副本数。
	- `AVAILABLE`：显示应用可供用户使用的副本数。
	- `AGE`：显示应用程序运行的时间。
	请注意期望副本数是根据`.spec.replicas`字段设置 3。

```bash
NAME               READY   UP-TO-DATE   AVAILABLE   AGE
nginx-deployment   3/3     3            3           2m39s
```

3. 查看上线状态，`kubectl rollout status deployment/nginx-deployment`
4. 查看Deployment创建的ReplicaSet(rs)，运行 `kubectl get rs`
	- ReplicaSet：输出中包含以下字段：
	- `NAME`：列出名字空间中ReplicaSet的名称。
	- `DESIRED`：显示应用的期望副本个数，即在创建Deployment时所定义的值。 此为**期望状态**。
	- `CURRENT`：显示当前运行状态中的副本个数。
	- `READY`:显示应用中有多少副本可以为用户提供服务。
	- `AGE`:显示应用已经运行的时间长度。
	注意ReplicaSet的名称格式始终为`[Deployment 名称]-[哈希]`。该名称将成为所创建的Pod的命名基础。其中的`哈希`字符串与ReplicaSet上的`pod-template-hash`标签一致。
	
```bash
NAME                          DESIRED   CURRENT   READY   AGE
nginx-deployment-86c57bc6b8   3         3         3       13m
```

5. 查看每个Pod自动生成的标签，`kubeclt get pods --show-labels`

```bash
NAME                                READY   STATUS    RESTARTS   AGE   LABELS
nginx-deployment-86c57bc6b8-lp6tg   1/1     Running   0          19m   app=nginx,pod-template-hash=86c57bc6b8
nginx-deployment-86c57bc6b8-w8j7s   1/1     Running   0          19m   app=nginx,pod-template-hash=86c57bc6b8
nginx-deployment-86c57bc6b8-x9r5r   1/1     Running   0          19m   app=nginx,pod-template-hash=86c57bc6b8
```
>必须在Deployment中指定适当的选择算符和Pod模版标签
>标签或选择运算符不要鱼其他空气之重叠，如果多个控制器具有重叠的选择算符，则控制器之间会冲突从而无法工作。
>不要更改pod-template-hash标签，此标签可确保子ReplicaSet不重叠
>
### 通过更新Deployment的PodTemplateSpec，声明Pod的新状态

>仅当Deployment pod模版(`.spec.template`)发生改变时，才会触发Deployment上线。其他更新(如对 Deployment 执行扩缩容的操作)不会触发上线动作。

更新当前Deployment
1. 更新Pod镜像`kubectl set image deployment/PodName-deployment Podimage=podimage:version`或者`kubeclt edit deployment/PodName-deployment`将`.spec.template.spec.containers[0].image`进行修改，输出`deployment.apps/nginx-deployment image updated`
2. 运行`kubectl get rs`以查看Deployment通过创建新的ReplicaSet并将其扩容到 3 个副本并将旧ReplicaSet缩容到 0 个副本完成了Pod的更新操作
更新Pod的时候只需要更新Deployment Pod模版即可，Deployment可确保在更新时仅关闭一定数量的Pod，默认情况下，确保至少所需Pod的75%处于运行状态(最大不可用比例为25%)
Deployment还确保所创建Pod数量只可能比期望Pod数高一点点，默认情况下，确保启动的Pod个数比期望个数最多多125%(最大峰值25%)

>Kubernetes在计算`availableReplicas`数值时不考虑终止过程中的Pod，`availableReplicas`的值一定介于`replicas - maxUnavailable`和`replicas + maxSurge`之间。 因此，你可能在上线期间看到Pod个数比预期的多，Deployment所消耗的总的资源也大于`replicas + maxSurge`个Pod所用的资源，直到被终止的Pod所设置的`terminationGracePeriodSeconds`到期为止。

多Deployment动态更新

当创建一个Deployment以生成`Nginx:1.22.2`的5个副本，但接下来更新副本创建`Nginx:1.28.2`，但此时只创建了3个`Nginx:1.22.2`副本，这种情况，会直接立即杀死旧的，并开始创建新版本的，不会等待旧的创建完成之后才开始执行变更操作

### 回滚 Deployment 版本

当Deployment不稳定，反复崩溃状态，可以使用Deployment保存在系统中的上线记录，进行回滚。可以通过修改修订历史记录限制记录约束。
比如当更新`Deployment`时写错了镜像

```bash
输入：kubectl set image deployment/nginx-deployment nginx=nginx:1.1xx   
输出：deployment.apps/nginx-deployment image updated
```

此时线上进程会出现停滞

```bash
输入：kubectl rollout status deployment/nginx-deployment
输出：Waiting for deployment "nginx-deployment" rollout to finish: 1 out of 3 new replicas have been updated...
```

查看创建的副本数及`pod`

```bash
输入：kubectl get rs
输出：
NAME                          DESIRED   CURRENT   READY   AGE
nginx-deployment-658bdfc99d   1         1         0       5m13s
nginx-deployment-7ccccd94f7   3         3         3       6m36s
 
输入：kubectl get pod
输出：
NAME                                READY   STATUS             RESTARTS   AGE
nginx-deployment-658bdfc99d-tfrdm   0/1     ImagePullBackOff   0          8m18s
nginx-deployment-7ccccd94f7-2twq8   1/1     Running            0          9m41s
nginx-deployment-7ccccd94f7-4mr2t   1/1     Running            0          9m41s
nginx-deployment-7ccccd94f7-kkkt5   1/1     Running            0          9m41s
```
`Deployment`控制器会自动停止有问题的上线过程，并停止对新的 ReplicaSet 扩容。 这行为取决于所指定的 rollingUpdate 参数（具体为 `maxUnavailable`）。 默认情况下，Kubernetes 将此值设置为 25%
检查Deployment上线历史

```bash
kubectl rollout history deployment/nginx-deployment

deployment.apps/nginx-deployment 
REVISION  CHANGE-CAUSE
1         <none>
2         <none>
```

查看指定版本的详情信息

```bash
kubectl rollout history deployment/nginx-deployment --revision=2
```

回退到上一个版本

```bash
kubectl rollout undo deployment/nginx-deployment
```

回退到指定版本

```bash
kubectl rollout undo deployment/nginx-deployment --to-revision=2
```

详情参考： https://kubernetes.io/zh-cn/docs/concepts/workloads/controllers/deployment/#rolling-back-a-deployment

### Deployment规模扩缩容

 手动扩缩容指令
 
```bash
kubectl scale deployment nginx --replicas=10
```

### 暂停、恢复Deployment的上线过程

在你更新一个 Deployment 的时候，或者计划更新它的时候， 你可以在触发更新之前暂停 Deployment 的上线过程。 当你准备应用这些变更时，你可以重新恢复 Deployment 上线过程。

暂停上线，暂停之后可以进行修改，且不会进行更新

```bash
kubectl rollout pause deployment/nginx-deployment
```

恢复 Deployment 上线并观察新的 ReplicaSet 的创建过程，其中包含了所应用的所有更新

```bash
kubectl rollout resume deployment/nginx-deployment
```

>你不可以回滚处于暂停状态的 Deployment，除非先恢复其执行状态

### 删除Deployment

使用`kubectl delete deployment/nginx-deploymane`删除即可，连同创建的pod
